# 多模态真实样本评测方案 v0.5

更新时间：2026-05-14

## 1. 评测背景

《听到了咩 / heard-sheep》当前已经具备：

- DeepSeek 文本任务分析能力；
- Xiaomi MiMo 图片理解 Provider；
- Xiaomi MiMo 音频理解 Provider；
- 图片入口和录音入口的真实能力验证链路。

但多模态能力是否足够进入长期主流程，不能只看单次调用成功。还需要用真实职场样本持续验证：

- 图片中文字识别的准确性；
- 音频转写的准确性；
- 多场景、多版面、多噪声下的稳定性；
- 输出是否适合作为用户确认页的初稿；
- 是否需要继续优化 Prompt、Provider，或引入专门 OCR / ASR 服务。

## 2. 图片评测范围

至少覆盖以下 6 类样本：

| 类型 | 示例 | 关注点 |
| --- | --- | --- |
| 微信聊天截图 | 领导文字交代、同事消息 | 对话顺序、发言人上下文、任务动作 |
| 飞书/钉钉消息截图 | 带多条消息 | 多条消息合并、时间和责任人 |
| 文档截图 | 需求说明、修改意见 | 段落顺序、修改项召回 |
| PPT 截图 | 汇报页中的行动项 | 标题、要点、行动项识别 |
| 表格截图 | 含时间、责任人、任务项 | 单元格结构、时间与责任人对应 |
| 会议纪要截图 | 多段连续文本 | 长文本顺序、待办与结论区分 |

建议每类至少准备 3 个样本，其中包含清晰样本、轻微压缩样本和复杂版面样本。

## 3. 音频评测范围

至少覆盖以下 6 类样本：

| 类型 | 示例 | 关注点 |
| --- | --- | --- |
| 清晰普通话短句 | 10 秒以内 | 基础转写准确性 |
| 清晰普通话长句 | 30-60 秒 | 长音频稳定性与断句 |
| 带口头语 | “那个、然后、你先……” | 口头语保留与可读性 |
| 带轻微环境噪声 | 空调声、键盘声 | 噪声下关键词召回 |
| 语速较快 | 连续交代多个任务 | 多动作识别与漏词 |
| 多任务混合交代 | 同一段中含多个行动项 | 时间、动作、对象的完整召回 |

可额外加入：

- 电脑麦克风录音；
- 手机录音；
- 会议外放录音；
- 不同距离和音量的录音；
- 含产品名、项目名、专有名词的录音。

## 4. 评测指标

### 图片评测指标

| 指标 | 说明 |
| --- | --- |
| provider 是否正确 | 应为 `xiaomi-image` |
| fallbackUsed | 应为 `false` |
| 是否返回非空文字 | `extractedText` 是否有效 |
| 关键任务信息召回 | 是否识别出时间、动作、对象 |
| 阅读顺序 | 截图多段文本是否顺序合理 |
| 可编辑性 | 是否适合作为“确认图片文字”初稿 |
| 误识别 | 是否出现明显凭空添加 |

### 音频评测指标

| 指标 | 说明 |
| --- | --- |
| provider 是否正确 | 应为 `xiaomi-audio` |
| fallbackUsed | 应为 `false` |
| 是否返回 transcript | 文本是否非空 |
| 关键词召回 | 时间、动作、对象是否听出 |
| 断句可读性 | 是否适合用户确认 |
| 口头语处理 | 是否保留得当 |
| 错听情况 | 专有名词、时间、核心名词是否出错 |

自动脚本只计算 provider、fallback、文本长度和关键词命中率；最终等级仍需人工判断。

## 5. 评测结论分级

```text
A：可直接进入主流程
B：基本可用，需用户小幅修改
C：可作参考，但需较多人工修正
D：不可用，需要 fallback 或替换 Provider
```

建议规则：

- A：关键时间、动作、对象完整，几乎无需修改；
- B：核心任务可用，存在少量错字或断句问题；
- C：能看出大意，但需要用户大幅修正；
- D：文本为空、fallback、关键信息错误或大量幻觉。

## 6. 后续决策规则

### 图片侧

- 如果多数样本达到 A/B：
  - 保留 Xiaomi 图片理解作为当前 MVP 方案。
- 如果聊天截图可用但表格 / 文档效果差：
  - 继续保留图片理解，但对复杂版面提示用户重点确认。
- 如果阅读顺序经常错乱：
  - 优先优化图片 Prompt，必要时拆分多图或引入版面分析。
- 如果整体 OCR 质量不稳：
  - 评估独立 OCR Provider。

### 音频侧

- 如果清晰语音达到 A/B：
  - 录音主链路可继续保留 Xiaomi 音频理解。
- 如果噪声 / 长音频效果明显下降：
  - 增加录音质量提示，并评估专门 ASR 服务。
- 如果关键信息常错：
  - 优先调转写 Prompt，再决定是否换 Provider。
- 如果大量样本 fallback：
  - 检查 API 配置、模型名、音频格式和 Base64 输入方式。

## 7. 推荐执行节奏

1. 每次 Provider 或 Prompt 调整后，至少跑一次 `npm run eval:multimodal`。
2. 对脚本输出的 JSON / Markdown 做人工复核。
3. 把人工等级填写到 `docs/多模态评测记录模板.md` 的记录表中。
4. 每累计 20 个真实样本，更新一次评测结论和后续决策。
